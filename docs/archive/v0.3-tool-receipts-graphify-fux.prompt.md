# Build prompt — graphify & fux token-saving receipts (hand to Claude Code)

> Copy everything below the line into Claude Code. You need the **fux** repo and
> the **cage** repo. **Do NOT touch the graphify repo — it is third-party and
> read-only.** Full spec: `cage/docs/tool-receipts.graphify-fux.handoff.md`.

---

Make graphify and fux feed token-saving receipts into cage so `cage attrib` /
`cage matrix` / `cage roi` show real with/without numbers for both. The with/without
*engines* already exist in cage; this is **instrumentation only**. The complete spec
is **`cage/docs/tool-receipts.graphify-fux.handoff.md`** — read it in full first; it
is the source of truth and every §-ref points into it.

**Two strategies, because the repos are owned differently (§1a):**
- **fux (yours):** in-tool fail-open shim — fux emits its own receipts.
- **graphify (third-party, READ-ONLY):** an external cage adapter wraps the
  unmodified `graphify` command and files the receipt by parsing graphify's output.
  **Zero edits to the graphify repo.**

## Non-negotiable constraints (violating any = failed change)

- **graphify repo gets ZERO diff.** All graphify measurement lives in cage. If you
  find yourself editing graphify, stop — you're doing it wrong.
- **cage is an OPTIONAL dependency for fux**, never hard. Do NOT add `cage` to fux's
  `pyproject.toml`. fux stays zero-runtime-dep. The fux integration is a fail-open
  lazy shim (§3): import cage inside `try/except`, no-op on any failure, so fux runs
  byte-identically with cage absent.
- **The cage adapter is transparent + fail-open.** `cage graphify -- graphify …`
  must pass graphify's stdout/stderr/exit through unchanged; a metering error never
  alters graphify's result. cage adapter is stdlib-only.
- **Deterministic, no model/network** on the measuring path. Token count is
  `round(len(text)/4)` (§3) — no tokenizer, no dependency.
- **`method="modeled"`, `confidence` per §4/§5** (~0.6 graphify, ~0.7 fux). The
  alternative is a reconstructed counterfactual — never `measured`.
- **Emit nothing** when `actual >= raw_alternative`; the graphify adapter also emits
  nothing when no `source_file` can be parsed/resolved (unmeasurable ≠ zero saving).
- **No PII / no bodies in `meta`** — only `op` + counts.

## Do, in order — stop if anything is red

**Stage 1 — fux shim.**
Add `fux/cage_receipt.py` (the exact §3 shim: `toks()` + fail-open `emit()`). Verify
it imports and no-ops with cage uninstalled.

**Stage 2 — fux instrumentation.**
At the hook-recall assembly point (`hooks`/`recall`), once the injected payload and
the selected source entries are known, compute `actual = toks(payload)` and
`raw_alternative = toks(relevant_sources_whole)` (§5 conservative default), then call
`cage_receipt.emit("fux", raw_alternative, actual, task=<session id>,
op="hook-recall")`. Same in `why`/`refs` only if low-cost.

**Stage 3 — graphify external adapter (in cage, no graphify edits).**
Add `cage/graphifymeter.py` + a `cage graphify` subcommand that:
- runs `graphify <args>` as a subprocess and passes stdout/stderr/exit through
  **unchanged**;
- captures the stdout answer → `actual = toks(answer)`;
- parses the answer for cited `source_file` paths — **prefer a structured mode**
  (`graphify … --json` if it exists) for a stable parse, else regex the text; pin
  the expected format in a comment;
- `raw_alternative` = sum of `toks()` over the **deduped, whole, present-on-disk**
  cited source files (bounded to touched files, never the repo);
- **emits nothing** if no source files parse/resolve, or if `actual >=
  raw_alternative`;
- otherwise files `cage.record_receipt(tool="graphify", unit="tokens",
  raw_alternative=…, actual=…, method="modeled", confidence=0.6,
  meta={"op": "<query|path|explain>"})`.
Wrap all metering in try/except so graphify's result is never affected.

**Stage 4 — tests (dependency-free).**
- fux: cage-absent no-op; one correct `tool="fux"` `modeled` receipt when present
  (stub `record_receipt`); no emit on non-saving; `meta` shape (criteria 2,3,4,7).
- cage adapter: `cage graphify -- …` stdout/exit byte-identical to bare graphify
  (criterion 1); one `tool="graphify"` receipt from a sample answer; emits nothing
  when the answer cites no resolvable file (criterion 4).

**Stage 5 — end-to-end verify + doc-sync.**
In a repo with cage installed and graphify available, run a real `fux hook-recall`
and a real `cage graphify -- graphify query …`, then from that repo's `.cage` root
run `cage attrib`, `cage matrix`, `cage roi`; confirm both tools appear as separate
rows with positive, additive, `modeled` savings and the graphify×fux 4-cell grid
(criterion 5). Doc-sync per §7 (cage README/agents.md; fux docs). **Do not write
docs into the graphify repo.**

## Working rules

- Plan before code: show me the §3 shim, the fux call site, and your graphify
  output-parsing approach (which mode, what regex/JSON path) **before** wiring emits.
- The graphify parser is the one fragile point — if neither structured nor text
  parsing reliably yields `source_file`s, **stop and ask**; do not guess the
  counterfactual (an inflated `raw_alternative` fabricates savings).
- Touch only what the spec names. No graphify edits, no new fux dependency, no
  drive-by refactors.

Acceptance = all 7 criteria in the handoff green, graphify repo unchanged (zero
diff), fux runs unchanged with cage absent, both tools appear in
`cage attrib`/`matrix`/`roi`, and no dependency added to fux.
