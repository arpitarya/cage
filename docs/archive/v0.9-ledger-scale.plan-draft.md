## 3.6 Ledger scale — partitions, scope, team aggregation

§3.1–3.5 each describe a single append-only file (`calls.jsonl`, `receipts.jsonl`,
`tasks.jsonl`, `provenance.jsonl`). That shape is correct for one developer on one
machine. Three pressures break the single-file assumption — and only these three,
so this section is deliberately bounded:

1. **Volume.** A heavy agent user emits 1–2k call rows/day. A single unbounded
   `calls.jsonl` makes every derive (`report`/`attrib`/`budget`) re-scan the whole
   history, even for `--since 7d`.
2. **Monorepo.** One `.cage/` at repo root covers many sub-projects. Today a row
   carries `session`/`task`/`agent` but no component key, so attribution cannot be
   sliced per sub-project.
3. **Team / shared view.** Each developer's ledger is gitignored and machine-local
   (correctly — see §3.6.3). Nothing combines them into a repo-wide number.

The fix for all three reuses idioms already in the plan: **partitioned files read
by glob** (a), **a counts-only optional key carried on the existing rows** (b), and
**`refs/notes` + merge-by-row-id, CI-sole-writer** (c) — the exact distribution
model §3.5 already uses for provenance.

### 3.6.1 Time-partitioned ledger files (read-path only)

The two pure-append logs — `calls` and `receipts` — are partitioned by UTC month:
instead of one `calls.jsonl`, the writer appends to `calls-YYYY-MM.jsonl`. Readers
`glob` the set and concatenate at read time — the in-memory row stream the derive
views see is unchanged. **`tasks.jsonl` is deliberately NOT partitioned** (it is
last-write-wins by `id`, §3.4 — see §3.6.1a).

```
.cage/ledger/
  calls-2026-05.jsonl   receipts-2026-05.jsonl
  calls-2026-06.jsonl   receipts-2026-06.jsonl
  tasks.jsonl           # single file — last-write-wins by id, low volume (§3.6.1a)
  provenance.jsonl      # local buffer (§3.5) — unpartitioned, short-lived, unchanged
```

- **Bounds re-scan.** Paired with the existing `constants.SINCE_WINDOW_DAYS`, a
  `--since 7d` query opens only the shards whose month overlaps the window, not a
  year of history. This is the whole point of the partition — `ledger.since_cutoff`
  already computes the cutoff; partitioning lets the reader *skip files* below it
  rather than filtering rows it already loaded.
- **Determinism preserved.** Shard filename derives from each row's own `ts`
  (already present), not a clock at write time. Same rows ⇒ same shards ⇒
  byte-identical reads. No new entropy.
- **Backward-compatible.** A legacy unpartitioned `calls.jsonl` is still globbed
  and read; migration is "new writes go to the dated file," never a rewrite of the
  append-only past (the ledger is never rewritten — CLAUDE.md invariant).
- **`provenance.jsonl` is exempt.** It is a buffer flushed to notes (§3.5), not a
  long-lived store, so it does not partition.

The partition is a *layout* change behind `ledger.append`/`ledger.read`; no derive
view, policy field, or schema field changes. The partition granularity
(`PARTITION_GRANULARITY = "month"`) lives in `constants.py` (the third audit layer)
so it is reviewable but not user-config.

#### 3.6.1a Why `tasks.jsonl` is NOT partitioned

`tasks.jsonl` is last-write-wins by `id` (§3.4), not pure append. Partitioning it
by month would let a task whose two writes straddle a month boundary (opened in
one month, closed in the next) land its rows in *different* shards — turning a
within-file last-write-wins resolution into a cross-shard timestamp reconciliation
the single-file form never needed. That is a correctness hazard imported to save
scanning a file that does on the order of hundreds of rows per year. So tasks stays
single-file; the writer partitions only `calls`/`receipts` (one `if kind in
("calls","receipts")` branch). Volume never justified partitioning tasks; its
update semantics actively argue against it.

#### 3.6.1b Reserve the `actor` identity field now

In the same schema pass that adds `scope` (§3.6.2), `schema.make_call` reserves an
optional `actor: str = ""` field (appended to `CALL_FIELDS`). It is **always empty
in v1** and PII-guarded — never a raw git author email/name, only a configured
handle or hash, validated like the other PII guards. The reason to add the field
now rather than when per-person rollup ships: the expensive, breaking part of
per-person attribution is the *contract* change (a field threaded through every
writer); the rollup itself is a cheap read-side addition. Paying the schema cost
once — while already touching the contract for `scope` — means per-person
attribution later is a read-side feature, not a second substrate migration against
a year of rows that never recorded an identity. The *rollup* (populating and
grouping by `actor`) remains deferred (§3.6.3).

### 3.6.2 The `scope` dimension (additive contract change)

To slice attribution per sub-project in a monorepo, calls and receipts gain one
optional field, `scope` — the **top-level changed directory** of the work, reusing
`tasks.jsonl`'s existing "top-level-dirs-only, never full paths" PII guard (§3.4).
It is *not* a path, not a module, not user-typed prose: it is the same coarse,
counts-safe partition key tasks already carry.

```jsonc
// calls / receipts gain one optional field
{ ..., "scope": "billing" }     // top-level dir; "" when unknown / non-monorepo
```

- **Contract change, so it goes through the substrate.** `schema.make_call` and
  `schema.make_receipt` gain `scope: str = ""`, appended to `CALL_FIELDS` /
  `RECEIPT_FIELDS`. Empty string is the default and the legacy/non-monorepo case —
  identical behavior to today when unset (no scope filtering applied).
- **Resolved the same way tasks resolve theirs.** When the meter knows the task's
  changed top-level dir (from the same git shell-out `tasks.py` already does,
  fail-open), it stamps `scope`. No new git code path; reuse `tasks`' resolver.
- **Read surface.** `report`/`attrib`/`budget`/`matrix` gain an optional
  `--scope <dir>` filter. No flag ⇒ byte-identical to today (the no-flag invariant
  §3.5 holds for `matrix --human` applies here too).
- **PII line unchanged.** Top-level dir only — never sub-paths, never file names.
  Narrower than provenance's repo-relative paths (§3.5), identical to tasks (§3.4).

### 3.6.3 Team aggregation via `refs/notes`, not a backend

**The ledger stays out of git, and stays machine-local.** `.cage/ledger/` is
gitignored by `cage init` and must remain so: it is high-churn append-only data,
and committing per-developer per-task cost into shared, permanent, undeletable git
*history* is a productivity-surveillance surface even with counts-only bodies. That
is the §3.5 reasoning ("counts never content," PII discipline) applied to spend.

But the team view does **not** require an external collector, object store, or
service — that would break the `$0`, stdlib-only, no-infra invariant that is Cage's
wedge. The team view reuses the **exact** §3.5 distribution model:

- **Buffer → notes.** Each machine's `.cage/ledger/` is the local buffer. A new
  `refs/notes/cage-ledger` ref is the canonical shared record, **written only by
  CI** (`CAGE_NOTES_WRITE=1`); a dev's `cage ledger-sync` defaults to a dry-run
  print of the merge plan, exactly like `cage notes-sync` (§3.5).
- **Single ref, merge by row id, never overwrite.** One `refs/notes/cage-ledger`
  holds mixed call+receipt rows — their `c_`/`r_` id prefixes make the type
  unambiguous and they never collide, exactly as the single `cage-provenance` ref
  holds all provenance rows. Splitting into per-type `cage-calls`/`cage-receipts`
  refs is a `# v2:` marker, warranted only if partial-fetch (pulling just spend or
  just savings — git notes fetch at ref granularity) is ever needed. The union
  itself reuses a **shared union-by-id helper** extracted from `notessync` — *not*
  `merge_rows` verbatim, whose collision tie-break ranks on `PROVENANCE_METHOD_TRUST`
  and whose `plan()` reads the provenance buffer by `sha`. Ledger rows use plain
  first-by-id (no method rank), since two machines only ever *add* unique ids,
  never edit a shared line. (This is why the "git merge hell" objection does not
  apply: union-by-id is a CRDT for append-only logs.)
- **Aggregate by `scope`, not by person, by default.** Default rollup dimension is
  `scope` (§3.6.2), not author — the shared artifact is a cost/ROI ledger, not a
  monitoring dataset. The `actor` field (§3.6.1b) is reserved in the contract now
  but **unpopulated and un-rolled-up in v1**; per-person attribution is a later
  opt-in read-side feature, never a default.
- **Read surface.** `report`/`attrib --team` reads the merged notes ref instead of
  the local buffer; no flag ⇒ local-only, byte-identical to today.

`cage verify`-style discipline carries over: the team read is **report-only**, never
a gate, and a missing/empty notes ref degrades to the local view (fail-open).

> **DECISION (changed from first draft, flag for review):** team aggregation uses
> `refs/notes/cage-ledger` rather than an external sink (S3/collector). Rationale:
> keeps `$0`/stdlib/no-infra; reuses the proven `notessync` merge-by-id law; the
> aggregate travels with the repo on clone (no setup) which is the adoption path
> that actually gets run. The earlier "external collector" idea is dropped. Veto
> point: if call/receipt volume per repo genuinely exceeds what notes should hold
> (single-digit GB/yr is fine; 100s of GB is not), revisit with an `export` shard
> to an out-of-repo store — but only then, and only with a named volume number.

### 3.6.4 Ledger-size warning (read-path, warn-only)

When a ledger grows past a threshold, a `ledger` read-path check sums the byte size
of the globbed shards and emits a single warning line to **stderr** — never stdout
(stdout is the deterministic table surface; a warning there would break byte-identity
and determinism). The threshold resolves policy-first: `policy.toml [ledger] warn_mb`,
else `constants.LEDGER_WARN_MB` (the `DEFAULT_CONFIDENCE`-style policy-preferred /
constant-fallback pattern). It is **warn-only and fail-open**: a `stat` error is
swallowed, the warning never raises and never blocks a write or a derive — the flux
invariant that deriving a report never refuses. The message names the size and the
remedy (archive old `*-YYYY-MM.jsonl` shards, or `ledger-sync` then prune), and
fires at most once per invocation. The default must sit well above a heavy solo-year
so it flags a genuine runaway, not normal power use — a flat low value trains users
to silence stderr. **Block mode is out** on the read/derive path; any write-path
disk-quota block is a separate future decision, not this one.

### 3.6.5 Invariants this amendment must not break

- **$0 / stdlib only** — no new dependency; `glob`, `datetime`, git shell-out only.
- **Determinism** — shard names derive from row `ts`; same ledger + policy ⇒ same
  tables. No clock/random on the read path.
- **Ledger never rewritten** — partitioning changes *new* write targets only; past
  shards are immutable; migration never rewrites history. Only `calls`/`receipts`
  partition; `tasks.jsonl` stays single-file (§3.6.1a).
- **PII** — `scope` is top-level-dir only; the reserved `actor` field is never a raw
  email/name (handle or hash only) and is empty in v1.
- **Four agents, always** — `scope` resolution and `ledger-sync` fan out to all of
  Claude Code · Codex · Copilot · Kiro; no surface is special-cased.
- **method is sacred** — unaffected; no cell's `measured/modeled/estimated` tag
  changes. Aggregation is a union of rows, not a re-derivation of method.
- **No-flag byte-identity** — `--scope`, `--team` all default off; output with no
  new flag is identical to pre-amendment output.
