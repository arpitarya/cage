# Task: org-deployed gateway — zero per-developer setup, all four agents

## Goal

Meter LLM token usage for **Claude Code, Codex, Copilot, and Kiro** across a whole
org with **no per-developer action** — no hooks, no MCP, no per-dev env edits, no
`cage meter --` wrapper. The org deploys one thing once (config pushed by MDM / a
shared gateway), and every developer's traffic is metered into a central ledger.
This is the "governance" tier above per-dev import/proxy.

Read `docs/cage-plan.md` §5 / §9.5 / §10, `CLAUDE.md`, and the existing
`cage/proxy.py` + `cage/usageparse.py` (the per-dev proxy this generalizes) before
designing. **Start by writing a design doc** (`docs/org-gateway.design.md`) and a
plan section; this is too large to land as one undocumented PR.

## Hard invariant

All four agents are first-class — Claude Code, Codex, Copilot, Kiro. A gateway that
only catches Claude/Codex is incomplete. See the per-agent reachability analysis
required below.

## The core tension — resolve it explicitly, don't hide it

cage law is **$0 / stdlib-only** on the default path. A *transparent* egress proxy
that tees HTTPS requires minting per-host leaf certs under a trusted root CA, which
stdlib can't do cleanly (no x509 issuance in `ssl`). So design in **two tiers** and
keep the core stdlib:

- **Tier A — managed explicit proxy (default, stdlib, $0).** A shared `cage gateway`
  process (generalize `cage proxy`: multi-upstream, multi-tenant). Developers reach
  it because MDM / managed device config pushes the base-URL + proxy env
  (`ANTHROPIC_BASE_URL`, `OPENAI_BASE_URL`, `HTTPS_PROXY`, …) and the gateway's TLS
  cert machine-wide — so it's zero *manual* setup, not zero deployment. Covers every
  client that honors base-URL/proxy env (Claude Code, Codex, and any others that do).
- **Tier B — transparent interception (opt-in extra `[gateway-mitm]`, NOT default).**
  For endpoint-pinned agents that ignore base-URL/proxy env (verify which — likely
  Copilot and Kiro), the only capture is transparent egress interception with an
  org-deployed trusted root CA. Fence this entirely behind an optional extra with its
  own cert-toolchain dependency, exactly like the `[embeddings]`/`[ml]` extras — it
  must never be imported on the $0 default path. Document the trust/security
  implications loudly.

The deliverable must state, per agent, which tier reaches it and why, grounded in
testing each client's behavior — not assumption.

## What to build

1. **`cage gateway`** — multi-upstream, multi-tenant evolution of `cage proxy`:
   - Route by upstream Host/SNI to the correct real API (Anthropic / OpenAI / GitHub
     Copilot / Kiro-AWS); parse `usage` per provider via `usageparse.py` (extend it
     for any provider shape it doesn't cover yet). Forward bytes verbatim;
     **fail-open and fail-fast** — a metering/parse error, or the gateway itself being
     degraded, must never change or block the developer's request. A gateway outage
     must fall back to direct egress, never wedge a dev's LLM call.
   - **Per-developer/team attribution** into the call row: derive a stable identity
     (hashed auth-token fingerprint, or an MDM-injected `X-Cage-Dev` header, or source
     identity) — store the **hash only**, never the raw token. Add the tenant/dev id
     as a first-class field via `schema.make_call` (update the contract + plan §3 if a
     new field is needed; keep it a closed/optional field).
   - **Central concurrent ledger.** Many devs write one ledger — make the append path
     safe under concurrency (advisory lock / per-writer shard files merged at read, or
     a single-writer queue). Preserve append-only + determinism of derived views.

2. **Per-agent reachability matrix** in the design doc: for each of the four — does it
   honor `*_BASE_URL`? `HTTPS_PROXY`? does it cert-pin? → therefore Tier A or Tier B.
   Test the real clients; cite what you observed.

3. **Doctor / ops surface.** `cage doctor --gateway` (or similar) reports gateway
   reachability and which agents are currently being captured org-wide. A central
   health endpoint that never serves request bodies.

4. **MDM deployment guide** (`docs/org-gateway.deploy.md`): the exact managed-config
   payload to push env + the gateway TLS cert (Tier A), and the root-CA + transparent
   route for Tier B, with the security review checklist.

## Constraints (cage law)

- **Additive, never a replacement.** The gateway is an org-governance option, not a
  successor to hooks/MCP/per-dev proxy. All existing metering paths keep working
  unchanged; a shop that wants hooks per-dev must be able to ignore the gateway
  entirely. Calls metered by both gateway and hook dedupe by id (no double-count).
- **$0 / stdlib only on the default path.** Tier A is stdlib. Any cert/MITM toolchain
  lives strictly in an opt-in extra, never imported by the core gateway.
- **Counts-never-content.** The gateway tees `usage` only — it must never persist
  prompt/response bodies, headers with secrets, or auth tokens. PII guard is stricter
  here than anywhere else in cage because it sees raw traffic. Bodies are read to
  extract token counts in-memory and immediately discarded.
- **Fail-open is safety-critical.** Unlike the per-dev proxy, this sits in the egress
  path for a whole org — degraded gateway ⇒ transparent passthrough / direct egress,
  with a metric, never a blocked or altered request.
- **Determinism.** Same recorded traffic + same policy ⇒ byte-identical derived
  tables, even with sharded concurrent writers.
- **All four first-class.** The routing table and tests enumerate all four agents.

## Acceptance

- Design doc with the per-agent Tier A/B reachability matrix, backed by observed
  client behavior.
- Tier A `cage gateway` implemented in stdlib: routes ≥2 upstreams, parses usage,
  tags per-dev identity (hashed), writes a concurrency-safe central ledger.
  `tests/` cover: multi-upstream routing; usage parsed per provider; identity hashed
  (raw token never stored — assert it's absent); concurrent appends don't corrupt or
  double-count; gateway-error path passes traffic through unmetered (fail-open).
- Tier B documented + scaffolded behind the opt-in extra, with the security/trust
  writeup — implementation may be phased, but the seam must exist and the core must
  not import it.
- `just test` green (112 passing); no existing plan-number assertion changes.
- Deployment guide an org admin could actually follow.

## Out of scope / explicit non-goals

No inspection or storage of prompt/response content, ever. No silent request blocking.
Don't make Tier B the default or let its dependency touch the core. If an agent
genuinely cannot be captured without endpoint changes you don't control, say so in the
matrix rather than claiming coverage.
