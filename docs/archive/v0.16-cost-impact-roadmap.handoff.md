# Handoff: Cage cost-impact roadmap — validate · diagnose · compare · estimate · verdict · study

**One-liner:** Make cage answer the product question end-to-end — *is this tool reducing my cost or increasing it?* — across all four agents on both CLI and VS Code, with measured stack comparisons, pre-task estimates calibrated against actuals, and a single verdict view.
**Owner / executor:** Claude Code
**Status:** Ready to build
**Stress-tested:** Devils-advocate challenged (a) scope soup → cut into 5 independently shippable phases, asks 1–2 reduced to validation of existing code; (b) `cage compare` as fake A/B → delta tagged `estimated` never `measured`, min-n gate, n+median+IQR always shown, no causal language; (c) thin-history estimates → band-not-point, min-n refusal, confidence comes from measured calibration; (d) untestable extensions → sanitized fixture corpus for CI, live checks manual with N/A. Pre-mortem top risk: wrong verdict on tiny n → min-n is a blocking acceptance criterion. Fleet-study debate (P5): (e) "refs/notes already does teams" → only for git-fluent teams; file bundles are the capture-only path, both kept; (f) week-vs-week is confounded by work mix → paired-by-machine deltas + caveat, phases recorded not remembered; (g) silent mid-week capture death is the #1 study-killer → coverage view gates conclusions, enrollment ends in `doctor`; (h) machine id is PII → opaque random id, mapping stays offline with the analyst. Residual risk: extension log formats may drift upstream — fixtures pin what we've seen, doctor flags what we haven't.

## 1. Context & background

Cage (a *flux*) already meters LLM traffic into an append-only ledger and derives spend/savings deterministically ($0, stdlib-only). The attribution engine (plan §4) answers "what did tool X save" via **modeled** marginal-by-fixed-order counterfactuals. What it cannot yet do:

- Prove capture actually works on every agent × surface combination (Claude Code / Codex / Copilot / Kiro, each as CLI **and** VS Code extension). Design says pull-based `cage import` is the universal path; the dummy-repo test plan (`docs/dummy-repo-test-plan.md`) exists but is manual and has not been executed against fixtures in CI.
- Show a **measured** comparison: "tasks run agent-only averaged 1k tokens; tasks run with agent+graphify averaged 500" — observed group totals, not modeled counterfactuals.
- **Estimate before, compare after**: predict a task's token/cost up front, record it, and report calibration (how often estimates land) as an empirical confidence level.
- Give the one-line answer: `cage verdict graphify` → saving or costing, composed from the views that already exist.

Existing assets this builds on: `attribution.py`/`matrix.py` (modeled counterfactuals), `tasks.py` (task rows, last-write-wins by id), `schema.py` (calls **and** receipts already carry `task` + `session` — no join field is missing), `quality.py` (`cage outcome`, `.cage/outcomes.json`), `forecast.py` (monthly projection — NOT per-task estimation), `regression.py` (cost drift), `roi.py`/`recommend.py`, `trend.py`, `debuglog.py` (CAGE_DEBUG observability, off by default), `doctorcmd.py` (`cage doctor`, exit-gates on fail), `errors.py` (single `CageError` at the read/CLI boundary), `transcript.py` (per-agent parsers: `parse_calls`, `parse_codex_calls`, `parse_copilot_calls`, `parse_kiro_calls`), `docs/dummy-repo-test-plan.md` + `docs/debugging-capture.md`.

## 2. Definition of done

- [ ] **P0** — fixture corpus in `tests/fixtures/transcripts/<agent>/<surface>/` (agent ∈ claude/codex/copilot/kiro; surface ∈ cli/vscode) with a parametrized test proving `cage import` parses every fixture to exact expected rows; a scripted dummy-sibling-repo scenario runner exists and passes (see §9).
- [ ] **P1** — `cage doctor --bundle` writes one redacted diagnostics archive (counts-never-content); a capture failure anywhere on the write path leaves at least one attributable line in `debug.log` when `CAGE_DEBUG=1` (audited by test).
- [ ] **P2** — `cage compare` groups completed tasks by stack signature and prints n / median / IQR tokens + USD per group and a delta row tagged `estimated`; refuses (with explanation) any group below `MIN_COMPARE_N`; byte-identical output on repeated runs over the same ledger.
- [ ] **P3** — `cage estimate` prints a band (median + IQR) from matching historical tasks, tagged `modeled`, refusing below min-n; the open task row can carry additive `est_tokens`/`est_usd`; `cage calibration` reports band hit-rate and ratio distribution over closed tasks with estimates.
- [ ] **P4** — `cage verdict <tool>` composes attrib + roi + trend + regression + quality into one verdict line plus its inputs, each input carrying its method tag; computes nothing new itself.
- [ ] **P5** — fleet study: rows carry an opaque additive `machine` id; `cage study start/stop <phase>` records phase markers; `cage export --study` emits a one-file bundle with manifest; `cage import <bundles…>` merges by id; `cage study report` shows per-machine coverage (gaps flagged) and **paired-by-machine** phase deltas (n/median/IQR, delta tagged `estimated`), refusing below `MIN_COMPARE_N`; enrollment is one command ending in a `doctor` check.
- [ ] Every phase: tests assert exact numbers; `just test` green; docs updated per §9.5.
- [ ] Nothing committed by the executor — all changes left in the working tree for Arpit's review.

## 3. Scope

**In scope:** the six phases above; new constants (`MIN_COMPARE_N`, `MIN_ESTIMATE_N`, band parameters) in `constants.py`; additive optional row fields (`label`, `est_*`, `machine`); new `cage query` explain entries for each new calculation; skillgen fragment updates for new subcommands.

**Out of scope (explicit):**
- Any server, telemetry endpoint, or auto-upload for fleet collection — bundles are plain files the user sends themselves (Slack/Drive/git — cage doesn't care).
- A controlled A/B experiment framework (randomized assignment, canaries) — v1 comparison is observational and says so.
- Any ML/embedding-based task-similarity matching — grouping is exact-key only (stack, scope, label).
- New capture mechanisms, proxies, or OS schedulers — capture stays as designed (hooks CLI-only; `cage import` universal).
- Rewriting `forecast.py` — monthly projection is a different question; leave it.
- Publishing/releasing — no version bump lands on PyPI from this work; **no commits at all** (Arpit reviews the tree first).
- Building a VS Code extension for cage itself.

## 5. Technical approach (decided)

### P0 — Validation harness (asks 1 & 2 verification)
1. **Fixture corpus.** `tests/fixtures/transcripts/{claude,codex,copilot,kiro}/{cli,vscode}/` — small sanitized session-log samples (token counts real, all content stripped). One parametrized test per fixture asserts the exact call rows `transcript.py` + `importcmd.py` produce (ids deterministic via `_composite_id`, so assertions are exact). Missing real samples are marked `OPEN QUESTION` (§10), with the CLI-format fixture standing in until captured.
2. **Dummy sibling repo runner.** A build-time-only script `tools/dummyrepo/run.py` (stdlib-only, never in the wheel — same rule as skillgen) that scaffolds `../cage-dummy-<ts>/` (or `--path`), runs the automatable steps of `docs/dummy-repo-test-plan.md` (init, wire status, plant fixture logs in the per-agent log locations via env-var overrides, `cage import`, derive views, assert exact numbers, PII grep of the ledger), and prints a pass/fail table. Manual extension steps remain manual; the runner prints them as a checklist. Update `docs/dummy-repo-test-plan.md` to reference the runner and add scenarios S-compare/S-estimate/S-verdict (§9).

### P1 — Diagnostics bundle (ask 2 completion)
`cage doctor --bundle [path]`: writes a single archive containing doctor output, `debug.log` + `hooks-seen.jsonl` (if present), `cage --version` + platform, resolved footprint paths, policy **provenance** (which file won, which env overrides set — never secret values), and cursor state. Counts-never-content is a hard gate: bundle contents pass the same PII discipline as the ledger. Plus a coverage audit test: monkeypatch each swallow-site on the capture path and assert `debuglog` receives an event — "fail-open but never silent" becomes tested, not aspirational.

### P2 — `cage compare` (ask 3: measured stack comparison)
- **Stack signature (derive-time, no schema change):** for each closed task, the sorted set of `tool` values on receipts whose `task` matches the task id (fallback join: receipts sharing the task's `session` within the task's time span). Empty set ⇒ `agent-only`. Tool receipts already exist (graphify/fux shims), so "with graphify" is visible in the ledger today.
- **Task totals (measured):** sum tokens/USD of calls with that task id (same session fallback; via `prices.call_usd` recompute). Group by `(stack, scope?, label?)`; `--scope`/`--label` filter, `--by` chooses grouping keys.
- **Output:** per group `n · median · IQR` for tokens and USD; a delta row between any two groups sharing all keys except stack, tagged `method=estimated` with a one-line caveat (`observed difference across different tasks — not a controlled experiment`). Groups with `n < MIN_COMPARE_N` (constants.py, default 5) render as `insufficient data (n=2 < 5)` — never a number.
- **Label:** additive optional `label` on the task row, set via `cage outcome <task> --label <word>`; same PII guard as `scope` (single token, no paths). Empty = legacy contract.

### P3 — `cage estimate` + `cage calibration` (ask 4)
- `cage estimate [--scope S] [--label L] [--agent A]`: from closed tasks matching the keys, print median and IQR of tokens and USD, tagged `modeled`, with n. Below `MIN_ESTIMATE_N` ⇒ refuse with the reason. Deterministic: same ledger ⇒ same estimate; no clocks in the math.
- `--record` stamps the estimate onto the **open** task row as additive `est_tokens`/`est_usd`/`est_n` (last-write-wins by id like every task field, fail-open write).
- `cage calibration`: over closed tasks carrying estimates, report actual/estimate ratio distribution, share landing inside the estimated IQR band, and a plain-English confidence line ("estimates landed in-band 78% of the time, n=41"). This measured hit-rate **is** the confidence level; the estimator itself never self-reports confidence.

### P4 — `cage verdict <tool>` (ask 5)
A pure composer: pulls marginal saving (attribution), net-of-own-cost (roi), direction (trend), drift (regression), redo-rate (quality), prints `VERDICT: graphify is SAVING ≈ $X/mo net (modeled)` — or COSTING / INSUFFICIENT DATA — followed by each input with its own method tag, and a break-even line (receipts/day needed for net-positive) derived from existing numbers. Hard rule: verdict computes no new statistics; if any input is unavailable it says so rather than approximating. Other §8-style ideas (per-feature rollup, carbon view) stay listed in the plan doc, not built.

### P5 — Fleet study (multi-laptop capture, one analyst)
The workflow: N laptops capture for a week agent-only, then a week with a plugin (e.g. graphify); one person produces the final number. Improvements this phase adds over raw export/import:

1. **Opaque machine id.** A random id generated once into `.cage/state/` (never hostname, username, or anything derivable — the analyst keeps the name↔id mapping offline). Stamped as an additive optional `machine` field on calls/receipts/tasks at write time; empty = legacy contract. Needed for cross-machine dedupe sanity, per-machine coverage, and paired deltas.
2. **Recorded phases, not remembered dates.** `cage study start <phase>` / `cage study stop` append a marker row (phase label = single validated token, same PII guard as `label`) to a small append-only `study.jsonl`. Derive assigns each call/task to a phase by its own `ts` against the markers — deterministic, no clocks at derive time. Phases beat receipt-inferred stacks here because a plugin can be installed but unused; the *intent* of the week is what's recorded. P2's stack signature remains the within-phase truth of what actually ran.
3. **One-file collection.** `cage export --study` writes a single bundle (rows + study markers + manifest: cage version, machine id, date span, row counts per kind — counts-never-content). The analyst runs `cage import bundle1 bundle2 …` into a fresh analysis ledger; merge-by-id makes re-imports idempotent. The refs/notes team path (§3.6) stays as-is for git-fluent teams; bundles are the low-friction fleet path.
4. **Coverage before conclusions.** `cage study report` opens with per-machine coverage: days with rows per agent per phase, gaps flagged (a laptop that went silent mid-week is the #1 study-killer). Then the number: **paired-by-machine** phase deltas (each machine's own baseline vs plugin week, then the median of paired deltas — controls between-machine variance), plus pooled n/median/IQR. Delta tagged `estimated` with the different-work-mix caveat; below `MIN_COMPARE_N` machines-with-both-phases ⇒ refuse.
5. **One-command enrollment.** `cage study join <phase>` = init/resolve global ledger → wire all four agents → start phase → run `cage doctor` and print the result + the cron line for `cage import`. A capture-only user runs exactly one command and can verify it worked.

**Decision order locked:** P0 → P1 → P2 → P3 → P4 → P5 (P5 reuses P2's grouping/refusal machinery; it lands last). P2–P4 each depend only on P0 fixtures for their scenario tests; P3 does not depend on P2's grouping code beyond shared key-matching helpers (factor into one small module, e.g. `cage/taskgroup.py`).

## 6. Non-negotiables / constraints

- **$0 / stdlib-only** on the default path; `dependencies = []` stays empty.
- **Determinism:** no clocks/random in any derived view (compare/estimate/calibration/verdict included); same ledger + policy ⇒ byte-identical output; tests assert exact numbers.
- **`method` is sacred:** group totals are `measured`; compare deltas and estimates are `estimated`/`modeled` and render their tag; nothing new may ever read as `measured` unless it is an invoice-grade recorded number.
- **Four agents, always:** every P0 surface and fixture set covers claude/codex/copilot/kiro; new work never special-cases one.
- **Two error regimes, never mixed:** write path stays fail-open (`--record`, bundle writes); read/CLI raises `CageError` only.
- **Additive-only schema changes:** `label`/`est_*` are optional task fields; empty = legacy contract; never a new required field, never a ledger rewrite. Update plan §3 alongside.
- **Numbers-layers:** `MIN_COMPARE_N`, `MIN_ESTIMATE_N`, band definition → `constants.py` with rationale comments; any user-tunable economics → policy.
- **PII:** labels are single tokens (validated), no paths/messages/content anywhere, bundle is counts-never-content.
- **Rendered assets:** new subcommands mean skillgen fragment edits + `python -m tools.skillgen && python -m tools.skillgen --bless` — never hand-edit the rendered skill files.
- **Do not touch:** `.github/workflows/publish.yml`, release flow, `metering.py`'s fail-open contract, provenance enums.
- **No commits, no releases:** leave everything uncommitted; version bump + CHANGELOG text prepared in the tree but not tagged/pushed/published.

## 8. Edge cases & risks

- Task with calls but no task id (transcript rows carry session only) → session-window fallback join; document the precedence in the module docstring and explain entry.
- Two stacks differing only by an incidental one-off receipt → stack signature uses tools with ≥1 receipt in that task; note in caveat line that signatures are per-task observed sets.
- A task spanning months (partitioned shards) → totals must read via `ledger.read_kind` globbing; test a cross-month task.
- Estimate recorded, task never closed → calibration skips open tasks; no error.
- Negative or zero actuals (task with no calls) → excluded from calibration with a counted "skipped" line, not silently dropped.
- Truncated ledger tail mid-compare → existing tolerant reader handles it; add one test.
- Upstream log-format drift (extension updates) → fixtures pin known formats; `cage doctor` already warns on unfireable hooks; a parse yielding 0 rows from a non-empty log writes a debug event (P1 audit covers it).
- Study: `start` without `stop`, or overlapping/repeated phase labels → last marker wins forward in time; report prints the resolved phase timeline so ambiguity is visible, never silent.
- Study: clock skew between machines → phases are resolved per-machine against that machine's own markers (both row `ts` and markers come from the same clock), so skew cannot cross-assign; note this in the module docstring.
- Study: a bundle imported twice, or two bundles from the same machine with overlapping spans → merge-by-id dedupes; S9 asserts idempotence.
- Study: rows predating enrollment (before any marker) → phase `none`, excluded from deltas, counted in coverage as unphased.
