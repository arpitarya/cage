# Handoff: visible source paths + per-source globs

**One-liner:** Make the built-in log-source registry **visible** in every project's
`policy.toml` as a generated, commented block, and let a `[sources]` entry declare its own
filename pattern — without moving the defaults out of code.

**Owner / executor:** Claude Code
**Status:** Ready to build. One release, two independent pieces (A and B below).

**Stress-tested:** Twice. The original ask — *"put all the paths in policy.toml, use only
from there"* — **failed its debate gate and was rewritten**, then a decision council
**killed the rewrite's Phase 2** and found a better design. Both passes are recorded because
the reasoning is the spec.

**Gate 1 (devils-advocate) — the freeze chain, each link verified in code:**
1. `initcmd.py:29` → `policy.default_toml()` is a verbatim `read_text()` of the bundle.
   Every `cage setup` freezes a copy of whatever the bundle ships.
2. `policy.load` merges `sources` at the **agent-table level** — a project `[sources.claude]`
   wholly shadows the bundled one.
3. `policysync._walk:126` **skips `"sources"` by design.** The one "bundle got newer"
   mechanism does not apply.
4. ⇒ A shipped `[sources]` default could never be fixed for an already-inited project.
   `paths.kiro_data_candidates()` is labelled `UNVERIFIED-LAYOUT` on Windows — a path fix is
   expected, not hypothetical.
5. The symptom is **silent zero capture**, not a visibly wrong number. Strictly worse than
   the stale-price class `policy_version` exists for.

**Gate 2 (decision council) — the crux that killed Phase 2:**
> *Does moving defaults from `.py` to `.toml` change who can edit them, or when a fix lands?*

No. `data/policy.toml` and `paths.py` **both ship inside the wheel**; the bundled policy is
**read-only at runtime** (CLAUDE.md). Both change only via a release + `pip install -U`.
Phase 2 bought **zero fix-latency**, and the user-override hatch (`[sources] … replace=true`)
already shipped in v0.28. Its entire delivered value was *visibility* — which its own
safety mechanism (a commented block) delivers alone, with no contract inversion and no
freeze risk.

**What survived:**
- **(A) A generated, commented `[sources]` block** in the bundled `policy.toml`, produced
  from `paths._builtin_log_sources()` by `tools/docgen`. Code stays the single source of
  truth; the block is inert documentation; CI gates drift. Delivers the original ask —
  paths with filenames, visible in `.cage/policy.toml`, resolved from one place.
- **(B) A per-source `glob` key.** The one real capability gap: `_GLOB_CHARS` (paths.py:376)
  rejects `*?[` in a policy path and `_FORMAT_GLOB` imposes the format's canonical glob, so
  a tool writing `usage-*.ndjson` is **uncapturable today**.

**Cut by the council, do not build:**
- Moving the defaults into the bundled `[sources]` table (old Phase 2).
- `${VAR:-default}` expansion — it existed *only* to express env homes in Phase 2's TOML.
  Phase 2 is dead; the justification died with it.
- Probing all OS candidates instead of `_first_existing` — fixes a hypothetical dual-install
  machine, breaks real byte-identity. Once (B) lands, such a user declares both sources.
- A `sources_version` in `[meta]`, a doctor pinned-source check, any `policysync` change.
  All were mitigations for a risk that no longer exists.

**Residual risk to watch (the council's minority report, preserved):**
1. `resolve_log_sources` keeps **two code paths** — a built-in branch and a policy branch.
   This design freezes that fork permanently. Accepted: unifying them was the only real
   cleanup Phase 2 offered, and it wasn't worth the freeze mode.
2. A generated comment block is **documentation that lies confidently** the moment docgen
   isn't run. This makes a 4th generated surface. CI `--check` catches drift — until
   someone ships with `--no-verify`.

**What would reverse this:** a real report of a non-standard install `[sources]` cannot
already express, or a vendor path move generating enough support load to justify
user-editable defaults. Neither exists today.

## 1. Context & background

Token counts are captured by sweeping each agent's session log. Candidate locations live in
two places:

- `paths._builtin_log_sources(agent)` — hard-coded `(dir, glob)` per agent, plus the per-OS
  helpers (`vscode_user_candidates`, `kiro_data_candidates`, `_first_existing`).
- `policy.toml [sources]` — user-owned additions/replacements (shipped v0.28.0,
  `docs/sources.md`).

`paths.resolve_log_sources(pol)` is **already** the single resolution point —
`importcmd`, `pathprobe` (`cage doctor --paths`) and `explain` all read through it. "Used
only from one place" is a property the code already has. What's missing:

- **You can't see the defaults without reading Python.** `cage doctor --paths` prints them,
  but `.cage/policy.toml` — the file you open to configure cage — says nothing about them.
  This is the complaint that started the work.
- **A policy source can't declare its own pattern.** `_FORMAT_GLOB[fmt]` is imposed and
  `_GLOB_CHARS` rejects any pattern in `paths`. A custom tool whose layout isn't
  `**/*.jsonl` is simply uncapturable.

## 2. Definition of done

**(A) Generated commented block**
- [ ] `tools/docgen`'s **existing `policy` target** (not a new one — see §5) emits a
      `[sources]` block into `cage/data/policy.toml`, between `# cage:sources-start` /
      `# cage:sources-end` sentinels, generated from `paths._builtin_log_sources()`.
- [ ] Every line of the block is a **comment**. `policy.load_project_raw()` on a fresh
      `.cage/policy.toml` returns **no** `sources` key.
- [ ] The block names, per source: the path, the glob (with what the pattern means), the env
      var that redirects it, the other OSes' locations, and the `UNVERIFIED-LAYOUT` label on
      Windows Kiro.
- [ ] `python -m tools.docgen --check` fails on drift; `--target policy` regenerates.
- [ ] Adding a source to `_builtin_log_sources` and *not* regenerating fails CI.
- [ ] Regeneration is **idempotent** and preserves the hand-written prose around the block.

**(B) Per-source glob**
- [ ] `[sources.<x>]` accepts an optional `glob`; absent ⇒ today's `_FORMAT_GLOB[fmt]`.
- [ ] A `[[sources.<x>]]` array-of-tables form works, each entry `{path, glob?}`.
- [ ] `path` still rejects glob chars — but the message names the fix (`put the pattern in
      glob =`) instead of dead-ending.
- [ ] `cage doctor --paths` shows the declared glob per source.

**Both**
- [ ] `tests/test_sources.py` passes **unmodified** (the additivity proof).
- [ ] Absent/empty `[sources]` ⇒ the built-in registry byte-for-byte.
- [ ] Capture identity: same files scanned, same rows, same ids on a default install.
- [ ] Derived identity: `cage report` / `cage insights attrib` byte-identical.
- [ ] Docs updated (§9.5).

## 5. Technical approach (decided)

**(A) Extend `gen_policy()`, do NOT add a `sources` target.** `TARGETS` maps
name → `(path, gen_fn)` and `main` writes each. A new `sources` target would write
`cage/data/policy.toml` — **the same file `policy` writes**. Two writers, one file, order-
dependent. Fold the sources block into the existing `gen_policy()` so `TARGETS` stays 1:1
with files and `--target policy` regenerates everything that file owns.

Sentinels, not the `FORMULA_ANCHOR` regex: that anchor is `<!-- formula: id -->`, a markdown
HTML comment. TOML needs `# cage:sources-start` / `# cage:sources-end`. Replace between
them; if they're absent, fail loudly rather than guessing where the block goes.

**(B) Both schema shapes, branch on parsed type.** `dict` ⇒ legacy table (`paths` +
`replace` + `format`, `_FORMAT_GLOB` imposed unless an optional `glob` is present);
`list` ⇒ array-of-tables, each `{path, glob?}`. TOML makes `[sources.x]` and `[[sources.x]]`
mutually exclusive per agent, which is why both shapes must be read.

**Why the block is commented, not active:** an active table would be copied verbatim by
`initcmd`, shadow the bundle via `policy.load`, and be un-upgradeable because `policysync`
skips `sources`. Inert = visible with none of that. This is the whole design.

**Why not just add a doc line instead:** the ask was to see the paths *in the file you
configure*. A generated block is that, and it can't drift.

## 6. Non-negotiables / constraints

- **$0 / stdlib only.** `dependencies = []` stays empty.
- **`tools/` is build-time only** — stdlib-only, never imported at runtime, never in the
  wheel. Same law as `skillgen`.
- **`paths.py` must not import `policy` at module scope** — `policy.py` does
  `from cage import paths`. Lazy-import in-function if needed (follow the existing
  `from cage import agents` cycle-breaker).
- **Fail-open.** A bad `[sources]` entry → `SourcesResolution.problems`, skipped. Never
  raise into a sweep.
- **Additivity is the contract.** Absent/empty `[sources]` ⇒ built-in registry byte-for-byte.
  Legacy `paths = [...]` + `replace` + `format` unchanged.
- **Determinism.** No clocks/random. Same policy + filesystem ⇒ same ordered source list.
- **Four agents, always** — claude · codex · copilot · kiro stay first-class in the block.
- **Generated assets are never hand-edited** — the block is docgen's, like `docs/formulas.md`.
- **Do not touch:** `policysync`'s `sources` skip; `_first_existing`; the
  `UNVERIFIED-LAYOUT` label (it must survive *into* the block); `LogSource` /
  `SourcesResolution` field names.
- **PII:** unchanged. Paths only, never content.

## 8. Edge cases & risks

- **Sentinels missing/corrupted** in `cage/data/policy.toml` → fail loudly with the
  regenerate command. Never guess an insertion point, never append a second block.
- **Idempotence:** running `--target policy` twice must produce identical bytes. The
  existing `gen_policy` strips prior `# formula: ` lines before rewriting — mirror that
  discipline for the sentinel block.
- **Home paths in a generated block.** `_builtin_log_sources()` returns *resolved absolute*
  paths for the current machine (`Path.home()` is baked in). The generated block must emit
  the **`~`-relative form**, or docgen output differs per developer and `--check` fails in
  CI. This is the single most likely way this change breaks. Derive the display form; do
  not `str(path)` the resolved value.
- **Env-set machines.** If a dev has `CLAUDE_CONFIG_DIR` set, `_builtin_log_sources()`
  returns the redirected path and the block generates wrong. Generation must read the
  **defaults**, not the env-resolved values — or docgen must be run with a clean env and
  that must be tested.
- **`glob` on a file source** (kiro) — ignored by `_scan`; the block must say so.
- Set-but-empty `glob = ""` → `problems`, not a silent `_FORMAT_GLOB` fallback.
