# Handoff: Ledger scale — partitions, scope, team aggregation (plan §3.6)

**One-liner:** Make the Cage ledger survive heavy/multi-dev/monorepo use via month-partitioned files (a), an optional counts-safe `scope` key (b), and `refs/notes`-based team aggregation reusing the provenance merge-by-id model (c).
**Owner / executor:** Claude Code
**Status:** Ready to build

## 1. Context & background
Each substrate file (`calls/receipts/tasks/provenance.jsonl`) is a single append-only log — fine for one dev, one machine. Three pressures break it: volume (heavy agent users emit 1–2k call rows/day, so every derive re-scans full history), monorepo (one root `.cage/` spans many sub-projects with no per-component key), and team view (machine-local ledgers never combine into a repo-wide number). The design-of-record amendment is `docs/cage-plan-3.6-ledger-scale.md` — read it first; it is the source of truth for this handoff. The chosen mechanisms deliberately reuse idioms already in the codebase rather than inventing new ones.

## 2. Definition of done
The build is done when ALL are true:
- [ ] Writers append to month-partitioned files for **calls and receipts only** (`calls-YYYY-MM.jsonl`, `receipts-YYYY-MM.jsonl`); readers glob + concatenate; legacy unpartitioned `*.jsonl` still read. **`tasks.jsonl` is NOT partitioned** (last-write-wins semantics — see §5(a)/§8).
- [ ] Shard filename derives from each row's own `ts`, never a write-time clock (determinism).
- [ ] `--since` queries skip shards whose month is entirely below the cutoff (not just filter loaded rows).
- [ ] `schema.make_call` / `make_receipt` carry optional `scope: str = ""`; `CALL_FIELDS`/`RECEIPT_FIELDS` updated; empty default = today's behavior.
- [ ] `schema.make_call` reserves an optional PII-guarded identity field `actor: str = ""` now (always empty in v1), so per-person rollup is a later read-side change, not a second contract change. Never a raw email; a configured handle or hash only (see §6 PII).
- [ ] `scope` is resolved from the existing `tasks.py` git shell-out (top-level dir only), fail-open, no new git code path.
- [ ] `report`/`attrib`/`budget`/`matrix` accept optional `--scope <dir>`; absent ⇒ byte-identical output to pre-change.
- [ ] `cage ledger-sync` exists, mirrors `cage notes-sync` structure: dry-run by default, writes `refs/notes/cage-ledger` only under `CAGE_NOTES_WRITE=1`; a shared union-by-id helper (extracted from `merge_rows`, not its provenance-method-coupled body) unions call/receipt rows.
- [ ] `report`/`attrib --team` read the merged notes ref; absent/empty ref degrades to local view (fail-open); default rollup dimension is `scope`, not per-developer identity.
- [ ] (d) When total ledger size (sum of globbed shard bytes) crosses a threshold, a one-line warning is emitted to **stderr** on read/derive — never stdout, never blocking, never raising. Threshold = `policy.toml [ledger] warn_mb`, falling back to `constants.LEDGER_WARN_MB`. Warning names the size and points at the remedy (archive old monthly shards / `ledger-sync`).
- [ ] All four agents (Claude Code, Codex, Copilot, Kiro) covered by any wiring touched; none special-cased.
- [ ] `just test` green; new tests for partition read/skip, `scope` round-trip + filter, ledger-sync dry-run/merge, `--team` fallback.
- [ ] Documentation updated to match (see §9.5).

## 3. Scope
**In scope:** the four sub-features (a) month partitioning of **calls/receipts only** + reserving the `actor` identity field, (b) `scope` field + filters, (c) `refs/notes/cage-ledger` team aggregation, (d) ledger-size warning, plus their tests and docs.
**Out of scope (explicit):** any external store / collector / service / S3 (rejected in §3.6.3 — would break $0/no-infra); partition granularities other than month; **partitioning `tasks.jsonl`** (last-write-wins, low volume — kept single-file); the per-person rollup *read* path and any opt-in UI (only the `actor` field is reserved now; populating + rolling up by it is v2); rewriting/migrating existing ledger history; partitioning `provenance.jsonl` (it's a short-lived buffer); changing any `measured/modeled/estimated` method tagging.

## 5. Technical approach (decided)
- **(a) Month partitions for calls/receipts; `actor` reserved; tasks exempt.** Writer picks `calls-YYYY-MM.jsonl` (and `receipts-…`) from the row's `ts`; reader globs all `calls-*.jsonl` (+ legacy `calls.jsonl`) and concatenates. `--since` uses `since_cutoff` to skip whole shards by month. `PARTITION_GRANULARITY="month"` in `constants.py`. **`tasks.jsonl` stays a single file** — it is last-write-wins by `id` (plan §3.4), and partitioning a task that spans a month boundary would split its two writes across shards and turn last-write-wins into a cross-shard timestamp reconciliation (a correctness hazard for a file that does ~hundreds of rows/yr). The writer takes one `if kind in ("calls","receipts")` branch; tasks is the cheap exemption. Also reserve `actor: str = ""` on `make_call` in the same schema pass so per-person attribution never needs a second contract change.
- **(b) `scope` additive field.** `scope: str = ""` on `make_call`/`make_receipt`, appended to field tuples; resolved via the `tasks.py` top-level-dir shell-out (fail-open). Read views gain optional `--scope`; default off = identical output.
- **(c) `refs/notes/cage-ledger` aggregation.** New `cage ledger-sync` mirroring `cage notes-sync`'s *structure* (dry-run default, `CAGE_NOTES_WRITE=1` to push). NOTE: `notessync.merge_rows` is not directly reusable — its collision tie-break ranks on `PROVENANCE_METHOD_TRUST` and `plan()` reads the provenance buffer keyed by `sha`. Extract the **pure union-by-id core** into a shared helper both paths call (provenance keeps its method-rank tie-break; ledger uses plain first-by-id, since call/receipt ids never legitimately collide). **Single `refs/notes/cage-ledger` ref** holding mixed call+receipt rows (ids are `c_`/`r_`-prefixed, so type is unambiguous and they never collide) — mirrors the single `cage-provenance` ref. Leave a `# v2:` marker: split into `cage-calls`/`cage-receipts` only if partial-fetch (fetching just spend or just savings) is ever needed — git notes fetch at ref granularity, so that's the trigger, and it's the same volume-veto territory as §3.6.3. `report`/`attrib --team` read merged ref, fall back to local. Default rollup by `scope`; per-person rollup is deferred (the `actor` field is reserved but unpopulated in v1).
- **(d) Ledger-size warning.** A `ledger` read-path check sums the byte size of the globbed shards and, if it exceeds the threshold, prints a single warning line to **stderr** (not stdout — stdout is the deterministic table surface; a warning there would break byte-identity tests and determinism). Threshold resolves policy-first: `policy.toml [ledger] warn_mb`, else `constants.LEDGER_WARN_MB` (the `DEFAULT_CONFIDENCE`-style policy-preferred-with-constant-fallback pattern). Warn-only and fail-open — a `stat` error is swallowed, the warning never raises, never blocks a write or derive. The message states the current size and points at the remedy (archive old `*-YYYY-MM.jsonl` shards, or `ledger-sync` then prune). Emitted at most once per command invocation, not per shard.

## 6. Non-negotiables / constraints
- **Style/patterns:** small single-purpose modules (fux spirit); stdlib only.
- **Use:** `glob`, `datetime`, git via shell-out (never `import git`), the shared union-by-id helper extracted from `notessync`, the existing `tasks.py` resolver. **Avoid:** any new dependency; any network/object-store/service; any `import`ed git library.
- **$0 / stdlib only:** `dependencies = []` stays. No new runtime dep.
- **Determinism:** no clock/random on read path; shard names from row `ts`; same ledger + policy ⇒ byte-identical tables. Tests assert exact numbers.
- **Ledger never rewritten:** new writes target dated files; past shards immutable; no migration that rewrites history.
- **`method` is sacred:** no cell's `measured/modeled/estimated` may change; aggregation is a row union, not a re-derivation.
- **Four agents always:** any wiring fans out to Claude Code · Codex · Copilot · Kiro.
- **No-flag byte-identity:** `--scope`, `--team`, partitioning must produce identical output to today when new flags are unused and only legacy files exist.
- **PII:** `scope` is top-level dir only — never sub-paths/filenames/prose. Team aggregate must not default to per-person identity. The reserved `actor` field must never carry a raw git author email/name — it holds a configured handle or a hash only, validated like the other PII guards, and is empty in v1.
- **Do not touch:** `provenance.jsonl` partitioning; the attribution method ladder; `policy.toml` schema; `est_cost_usd` derive-time fallback logic in `prices.py`.

## 7. Dependencies & prerequisites
- Git available for `scope` resolution and notes sync (already assumed by §3.4/§3.5; fail-open when absent).
- Env var (by name only): `CAGE_NOTES_WRITE` — gates the real notes push; CI sets it, dev defaults to dry-run.
- No new services, credentials, or external storage.

## 8. Edge cases & risks
- **Month boundary (calls/receipts):** a row with a `ts` in a different month than "now" lands in its own dated shard — correct and required for determinism; tests must cover a cross-month ledger. This is also exactly why `tasks.jsonl` is NOT partitioned: a task whose two last-write-wins writes straddle a month boundary would split across shards, forcing cross-shard timestamp reconciliation. Keeping tasks single-file avoids that entirely.
- **Legacy + partitioned coexistence:** an existing `calls.jsonl` alongside `calls-2026-06.jsonl` must both be read, deduped only if a true id collision (there shouldn't be one).
- **Truncated tail shard:** read tolerance must hold per-shard (last partial line ignored), as today.
- **Non-repo / detached HEAD:** `scope` resolution and `ledger-sync` omit/skip and never raise (fail-open).
- **Empty/missing notes ref:** `--team` degrades to local view, no error.
- **Disagreeing fragments on merge:** two machines never edit the same id; if an id somehow collides, keep first-by-id-order (document the rule), never silently overwrite.
- **Risk — notes volume:** call/receipt volume is higher than provenance. Single-digit GB/yr is fine for notes; if a repo blows past that, the amendment's veto point (out-of-repo export shard) applies — leave a `# v2:` marker, do not build it now.
- **Size warning must not perturb output:** the warning goes to stderr only; stdout tables stay byte-identical, so the determinism + no-flag-byte-identity tests must run with stderr ignored and still pass. The warning must fire at most once per invocation (not once per globbed shard).
- **Size warning fail-open:** a missing/unreadable shard during the byte-sum is skipped, never raised; if the threshold can't be resolved, default to `constants.LEDGER_WARN_MB` and continue.

## 10. Decisions & open questions

**DECIDED (debated, folded into the spec above):**
- **Team identity** → scope-only rollup in v1, BUT reserve the PII-guarded `actor` field on `make_call` *now* so per-person attribution is a later read-side change, not a second contract change. (The debate point that held: schema retrofit is the expensive part, so pay it once while already touching the schema; the read-side rollup is the cheap part to defer.)
- **Partition tasks** → NO. Partition calls/receipts only; `tasks.jsonl` stays single-file. Its last-write-wins semantics + month-straddling tasks make partitioning a correctness hazard, and its volume never justified it. (Verdict flipped from the original "all three.")
- **Notes ref** → single `refs/notes/cage-ledger` (mixed `c_`/`r_` rows), mirroring `cage-provenance`; `# v2:` marker to split into per-type refs only if partial-fetch is needed. (Split now = over-engineering for the volume case §3.6.3 already defers.)
- **`cage query` concept** → fold *aggregation* into the existing `data-flow` entry; add NO standalone partitioning concept (invisible layer + per-entry maintenance tax). (Recommended pick was half-right; trimmed.)

**STILL OPEN:**
- OPEN QUESTION: the size threshold needs a *derived* basis, not a round number. The debate killed a flat 50 MB (fires constantly for power users → trained to `2>/dev/null`). Proposal: warn when **total bytes exceed K × the largest single shard's expected monthly size**, or simply set `LEDGER_WARN_MB` well above a heavy solo-year (≈250–500 MB) so it only flags a genuine runaway/shared-dir accident. Pick the basis. **Block mode stays out on the read/derive path** (flux invariant — deriving a report never refuses); a write-path disk-quota block is a *separate* future decision, not this one.
