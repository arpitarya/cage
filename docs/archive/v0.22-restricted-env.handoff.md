# Handoff: restricted environments — no-exe launcher mode + zipapp distribution

**One-liner:** Make cage first-class on locked-down (finance/enterprise) Windows endpoints where unknown exes are blocked: an opt-in python-launcher wiring mode that never invokes anything exe-shaped, a CI-built single-file `cage.pyz` (stdlib zipapp) release asset for machines without pip/PyPI access, and a "restricted environments" doc tier.
**Owner / executor:** Claude Code
**Status:** Ready to build
**Stress-tested:** (a) "a .pyz is a second distribution channel — release complexity" → held partially: built by a separate CI job attached to the same GitHub release, checksummed, and the PyPI trusted-publishing job is untouched — one flow, one extra artifact; (b) the real technical landmine: cage reads bundled data via `Path(__file__).parent / "data"` (`paths.bundled_data_dir`), which **breaks inside a zipapp** — migrating bundled-data access to `importlib.resources` is a prerequisite task, not a footnote; (c) "python-launcher mode fragments wiring" → mode is a persisted project-wide setting, idempotent across re-runs, reported by doctor — never a per-file drift; (d) "unsigned .pyz has the same trust problem as an exe" → materially different: endpoints block unknown *executables*; a .pyz runs through the already-approved Python interpreter, and the sha256 in the release notes gives compliance one auditable file. Residual risk: some WDAC policies also constrain script hosts — documented honestly as "check your policy", not promised around.

## 1. Context & background

Cage on Windows currently works via the pip-generated `Scripts\cage.exe` entry point (a launcher exe requiring installed Python). Finance/enterprise endpoints (AppLocker/WDAC) commonly block unsigned or non-allowlisted exes — including pip entry-point launchers in user-writable paths. Cage never structurally needs the exe (`py -m cage` is fully equivalent, and the `cage-run` shim already falls back to it — see `docs/portable-wiring.md`), but two gaps remain:

- The shim **prefers** the exe: on a machine where AppLocker *finds* `cage.exe` but blocks executing it, the shim surfaces a blocked-execution error on the hook path instead of falling through to `py -m cage`.
- Installation itself may be blocked: no PyPI access, pip restricted. Cage's stdlib-only/zero-dep design makes a single-file stdlib `zipapp` (`cage.pyz`, run as `py cage.pyz …`) a near-free distribution answer — but only after the bundled-data access pattern survives running from inside a zip.

The third tier — internal Artifactory/Nexus mirror — is documentation only (cage's `dependencies = []` and OIDC trusted publishing are the review answers).

## 2. Definition of done

- [ ] **Python-launcher mode:** `cage setup --python-launcher` (persisted, e.g. `[wiring] python_launcher = true` in project policy) writes shims + user-level wiring that invoke `py -m cage`/`python3 -m cage` directly and never probe or execute `cage`/`cage.exe`; runtime override `CAGE_RUN_PYTHON=1` makes an existing standard shim skip exe probing. Idempotent; re-running setup preserves the mode; `cage doctor` reports which mode is active.
- [ ] **Bundled data works from a zip:** all bundled-data reads (`data/policy.toml`, skills/prompts/steering assets, graphify shim source) go through `importlib.resources` (stdlib) and are tested under a built `.pyz`.
- [ ] **`cage.pyz` release asset:** a CI job (separate from the PyPI publish job, same `release: published` trigger) builds `cage.pyz` via stdlib `zipapp`, smoke-tests it (`py cage.pyz --version`, `report`, `demo` on the runner matrix), and attaches it + a `SHA256SUMS` file to the GitHub release. The PyPI trusted-publishing job is not modified.
- [ ] **`.pyz` parity honesty:** `cage --version`/`doctor` under the pyz identify the distribution (`cage X.Y.Z (zipapp)`); anything that genuinely can't work from a zip (if found) is documented and doctor-reported, never silently broken.
- [ ] **Docs:** `docs/restricted-environments.md` (three tiers: python-launcher mode · cage.pyz · internal mirror; what each solves; the WDAC script-host caveat) linked from README's install/platform section; `cage query restricted-env` concept entry.
- [ ] Tests green (`just test`), dummyrepo scenario extended (launcher-mode wiring greps + pyz smoke), skillgen `--check` clean.
- [ ] Nothing committed — working tree only, for review.

## 3. Scope

**In scope:** the launcher mode, the importlib.resources migration, the pyz CI job + smoke tests, docs/query entries, doctor reporting.
**Out of scope (explicit):** any frozen/compiled binary (PyInstaller etc.) — worse for this threat model; code signing infrastructure; an MSI/installer; changing the default wiring mode (standard shim stays default); touching the PyPI publish job's steps.

## 4. Current state

- Repo: cage @ v0.20.x. Key files: `cage/runshim.py` (shim writer, both variants), `cage/paths.py` (`bundled_data_dir` — the `__file__`-based pattern to migrate), every consumer of `bundled_data_dir` (grep it), `cage/setupcmd.py` + `agents.py` (setup flow, user-level wiring), `cage/doctorcmd.py`, `.github/workflows/publish.yml` (trigger reference only — do not modify its publish job), `docs/portable-wiring.md` (design of record for the shim — this work extends it and must update it).

## 5. Technical approach (decided)

1. **Launcher mode** rides the existing shim machinery: same files, same fail-open exit-0 contract, different resolution body (straight to `py -m cage` / `python3 -m cage`). Persisted project-wide; user-level (per-machine) wiring written by setup honors the mode too. `CAGE_RUN_PYTHON=1` is the no-rewire escape hatch checked at shim runtime.
2. **importlib.resources migration** first (it's the pyz prerequisite): one helper replacing `bundled_data_dir`, `as_file` where a real filesystem path is unavoidable (e.g. copying skill assets out). Byte-identical behavior installed-from-wheel (tested).
3. **zipapp build:** stdlib only — stage the package, `python -m zipapp` with `main="cage.cli:main"`, no compression exotica. CI job on the release trigger; artifacts: `cage.pyz` + `SHA256SUMS`. Smoke matrix reuses the existing 3-OS runners.
4. **Doctor/version awareness:** detect zip-origin (`__file__` inside an archive) → label the distribution; portability check unchanged.

## 6. Non-negotiables / constraints

- $0/stdlib (`zipapp`, `importlib.resources` are stdlib); determinism untouched (this is packaging/wiring, not data); fail-open shim contract preserved in both modes; four agents (launcher mode covers all four's wiring); additive-only — standard mode remains the default and byte-identical for existing users.
- **Do not modify the PyPI publish job** — the pyz is a sibling job on the same trigger. Never publish from local; the pyz is CI-built only.
- No new wheel contents (tools/ stays out); no signing promises — checksum + interpreter-mediated execution is the honest offer.
- Rendered skill assets via skillgen fragments only.
- No commits/tags/pushes/publishes by the executor.

## 7. Dependencies & prerequisites

None external. A real locked-down Windows endpoint for field validation is OPEN QUESTION territory (same posture as the Windows manual checklist — CI-tested until a fleet participant validates).

## 8. Edge cases & risks

- `importlib.resources` on a namespace/zip edge: test every bundled asset read under the pyz, not just policy.toml.
- `cage setup` from inside a pyz writing skill/shim files: must extract via `as_file`, and written shims must still resolve a *normal* cage at runtime (the pyz user may have no `cage` command — launcher-mode shims under pyz should resolve to `py <path-to->cage.pyz`? **No** — decided: shims never embed the pyz path (machine-specific); document that hooks + pyz = run `cage setup --python-launcher` only if the package is also importable, else hooks stay unwired and pull-based `py cage.pyz import` is the documented path. The pyz's primary story is capture-by-import, not hooks.)
- WDAC policies that constrain script hosts: documented caveat, doctor can't detect it — don't pretend.
- `py` vs `python3` naming per OS in launcher mode: reuse the shim's existing per-OS twins.
- Release asset upload failure must not fail the PyPI publish (independent jobs).

## 9. Testing & validation

Unit: importlib.resources helper (wheel + pyz contexts); launcher-mode shim contents (grep: no `cage.exe`, no bare `cage` probe); mode persistence + idempotency; doctor mode/distribution reporting. Integration: build pyz in CI and locally (`python -m zipapp` smoke in dummyrepo scenario): `--version`, `demo`, `import` over a fixture log, `report` byte-identical to wheel-installed output on the same ledger. Manual (fleet participant, when available): the restricted-environments doc's own checklist on a locked-down endpoint.
