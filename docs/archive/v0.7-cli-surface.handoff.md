## 0. What this is (and what it deliberately is NOT)

A review of all ~30 `cage` subcommands asked the obvious question: *is this too many,
what can be merged?* The broad answer — collapse ~30→16 by folding `roi`→`attrib`,
`trend`→`human`, and the §8 cluster into `cage ops` — **was considered and rejected.**
This handoff implements only the part that survived an adversarial review.

**Rejected, on purpose (do NOT implement; recorded so it isn't reopened):**

- **`roi`→`attrib --roi`, `trend`→`human --over`, `quality`/`regression`/`recommend`/
  `forecast`→`cage ops`.** Reasons: (1) each verb maps 1:1 to a numbered plan section
  (§8.x) with a `query` explainer — that spec→verb→formula traceability *is* the audit
  story; nesting muddies it. (2) Cage is agent-as-user (principle 6); flat verbs
  enumerate and call more reliably than `--view`/noun-nested forms, and `mcpserver.py`
  maps MCP tools to these verbs — merging splits the CLI and MCP surfaces. (3) Zero new
  capability; pure refactor risking regression across 112 tests. 30 flat verbs is not a
  crisis at current scale (git ships ~150). Revisit only if cold-user `--help`
  confusion becomes a measured problem.

**Approved (this handoff):**

- **A. Collapse the setup cluster** — `init` + `setup` are the front doors; `adopt` and
  `hooks` become thin deprecated aliases that forward to `setup`. This is the one real
  failure: `init ⊂ adopt ⊂ setup` and `hooks` is their wiring slice — nested subsets a
  user can't choose between. Onboarding is the adoption funnel.
- **B. Hide the 5 internal `hook-*` entrypoints** from `--help`. Free, no back-compat cost.

## 1. Laws this must respect (non-negotiable)

- **No breaking changes to public verbs.** `adopt`/`hooks` keep working — as aliases,
  not errors. Anton or any CI may call them. A removed verb is a one-way door; a
  deprecation alias is not.
- **Idempotency.** All wiring (`agents.install`, graphify interceptor, hook install) is
  idempotent today — keep it. Routing `adopt`→`setup` must not double-write hooks.
- **$0 / stdlib-only.** argparse only. No new deps.
- **Determinism / fail-open** on any read path touched.
- **One front door, documented.** After this, `cage --help` should make it obvious:
  `init` = scaffold, `setup` = wire an agent. No third "which do I run?" verb in the
  visible surface.

## 2. Part A — collapse the setup cluster

### 2.1 Target surface

| verb        | role after change                                                        |
|-------------|--------------------------------------------------------------------------|
| `init`      | **keep, unchanged.** Scaffold `.cage/` (policy + gitignored ledger). The primitive. |
| `setup`     | **the one front door.** Interactive wizard, or non-interactive via `--<agent>`. Absorbs adopt + hooks via flags (below). |
| `adopt`     | **deprecated alias** → forwards to `setup` with project-scaffold defaults; prints a one-line deprecation notice to stderr. |
| `hooks`     | **deprecated alias** → forwards to `setup` in wire-only mode; one-line deprecation notice. |
| `doctor`    | **keep, unchanged.** Verify a project's setup.                            |

### 2.2 What `setup` must cover (so adopt/hooks have a home)

Today: `setup` = skill + init + per-project wiring + graphify, interactive or `--<agent>`.
`adopt` = init + graphify + PATH, opt-in agent wiring. `hooks install|status` = agent
wiring only. Fold those two capabilities into `setup` as explicit modes:

- `cage setup --project-only` → what `adopt` did: per-project scaffold + graphify
  interceptor + PATH, agent wiring opt-in via `--<agent>`. (i.e. skip the global skill.)
- `cage setup --wire-only [--<agent>]` → what `hooks install` did: agent wiring only,
  no scaffold, no graphify.
- `cage setup --status` → what `hooks status` did: report current wiring, change nothing.

Keep the existing `--no-skill` / `--no-project` / `--no-graphify` / `--<agent>` flags.
`--project-only` is sugar for `--no-skill`; if that equivalence is exact, implement
`--project-only` as an alias of `--no-skill` rather than a new branch.

### 2.3 Deprecation aliases (back-compat, one release)

Implement `adopt` and `hooks` as **hidden** subparsers that forward:

```python
# adopt → setup --project-only  (+ pass through --<agent>, --no-graphify)
def cmd_adopt(args) -> int:
    _warn_deprecated("adopt", "setup --project-only")
    return cmd_setup(_as_setup_args(args, project_only=True))

# hooks [install|status] → setup --wire-only [--status]
def cmd_hooks(args) -> int:
    _warn_deprecated("hooks", "setup --wire-only")
    return cmd_setup(_as_setup_args(args, wire_only=True, status=(args.action == "status")))
```

- `_warn_deprecated` prints `cage: 'adopt' is deprecated, use 'cage setup
  --project-only'` to **stderr** (not stdout — don't corrupt `--json`/piped output), once.
- Register both with the help-suppression mechanism in Part B so they vanish from
  `--help` but still parse. Keep them at least one tagged release; removal is a separate
  PR with its own changelog line.

## 3. Part B — hide internal entrypoints from `--help`

The 5 `hook-session-start` / `hook-session-end` / `hook-post-tool-use` /
`hook-post-commit` / `hook-prepare-commit-msg` entries are wired by `cage hooks install`
and never typed by a human. Plus the two new deprecated aliases. Hide all of them from
the help listing without breaking dispatch.

**Mechanism (argparse, stdlib):** set a curated metavar on the subparsers so argparse
stops auto-printing the full choices enumeration, and let the existing `_EPILOG`
category block be the canonical command list:

```python
sub = p.add_subparsers(dest="cmd", required=False, metavar="<command>")
```

(`required=False` is also needed for the bare-`cage` banner in the other handoff — keep
them consistent.) The `_EPILOG` already lists the real commands by category; ensure the
internal `hook-*` and the deprecated `adopt`/`hooks` are **absent** from that epilog. Net
effect: `cage --help` shows only the curated category list; the hidden verbs still parse
and run.

If a future argparse makes per-subparser `help=SUPPRESS` reliable, prefer that for the
deprecated aliases so they're individually suppressible — but the metavar approach is the
portable floor.

## 5. Acceptance criteria

1. `just test` green (112 + new).
2. `cage --help` lists only the curated categories — no `hook-*`, no `adopt`, no `hooks`.
3. `cage init` unchanged (byte-identical output test).
4. `cage adopt --claude` and `cage hooks install --claude` still **work**, produce the
   same wiring as before, and each print exactly one deprecation line to **stderr** (assert
   stdout is unchanged vs. the equivalent `cage setup` invocation).
5. `cage setup --project-only` reproduces the old `adopt` result; `cage setup --wire-only`
   reproduces old `hooks install`; `cage setup --status` reproduces old `hooks status`.
6. Idempotency: running `cage adopt --claude` then `cage setup --project-only --claude`
   does not double-write hooks or duplicate PATH/graphify entries.
7. `cage --json` / piped output is uncorrupted by the deprecation notice (it's on stderr).
8. `cage doctor` still passes on a project set up via the new `setup` modes.
