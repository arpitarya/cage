# Handoff: `cage policy sync` — upgrade a project policy.toml to the installed bundle

**One-liner:** A `cage policy sync` command (dry-run by default, `--apply` to write) that brings a project's `policy.toml` up to the installed bundle: adds new sections/props, updates stale un-customized defaults, never touches user-customized values, warns on dropped keys — with prices/credits/alias/tools tables delegated to the existing `prices sync` machinery.
**Owner / executor:** Claude Code (cage repo)
**Status:** Ready to build
**Stress-tested:** (a) name `cage sync` → rejected: `ledger-sync`/`notes-sync` already mean *distribution*; a third unrelated "sync" overloads the word — `cage policy sync` (with room for `policy diff`) wins, and hints print the full command so discoverability doesn't suffer. (b) "auto-apply on upgrade/import" → rejected: never auto-applied is the standing law from prices sync; hints recommend, humans run. (c) "just extend `prices sync --all`" → rejected: prices sync's contract is pricing tables; general props deserve their own verb, which *delegates* to prices sync for the table families instead of duplicating its merge brain. (d) pre-mortem top failure — sync silently flips a behavior toggle → killed by customized-detection (only values equal to their old bundled default update) plus the byte-identical invariant test below. Residual risk: old-bundle reconstruction needs `[meta] policy_version` history; where the old default isn't reconstructable the key falls to the "listed, per-row confirm" bucket — honest over clever, same as prices sync.

## 1. Context & background

A project inited at v0.16 has a policy.toml missing everything added since: `[cleanup]`, `[capture] import_before_export`, `[wiring] python_launcher`, `[human] idle_cap_minutes`, `[prices] stale_days`, new `[meta]` fields. `policy.load` defaults all of them, so nothing breaks — but the user never *discovers* tunables, and a stale un-customized default can drift from the bundle's improved one. `cage prices sync` solved exactly this for pricing tables (customized-detection via `prices_version`, dry-run default, managed writes). This generalizes it to the whole file.

## 2. Definition of done

- [ ] `cage policy sync` (dry-run: categorized diff, no writes) and `cage policy sync --apply`; `cage policy diff` as an alias for the dry-run view.
- [ ] Four categories, each rendered under its own heading with counts:
      1. **add** — sections/keys in the bundle, absent in the project → written with bundled defaults into cage-managed placement, one provenance comment (`# added by cage policy sync (vX.Y)`).
      2. **update** — project value == its *old* bundled default (reconstructed via `[meta] policy_version`) and the bundle's default changed → updated, old→new shown.
      3. **keep (customized)** — project value differs from its old bundled default → never touched, listed so the user sees what sync deliberately skipped.
      4. **orphan** — project keys the new bundle no longer ships → warned with context, never deleted.
- [ ] Pricing-family tables (`[prices]`, `[credits]`, `[alias]`, `[tools]`) delegate to the `prices sync` implementation (one merge brain); the policy-sync output includes its summary rather than re-diffing those tables.
- [ ] `[meta]` gains `policy_version` (bundled, stamped on init and on every `--apply`) alongside `prices_version`; unknown/absent project `policy_version` (pre-this-feature projects) → adds still work, updates fall to per-row confirmation (`--yes` accepts all shown), matching the prices-sync fallback.
- [ ] **Behavior-neutrality invariant, tested:** on a project with zero customizations, `--apply` then every derived view ⇒ byte-identical output to before (adds only pin defaults `policy.load` was already using).
- [ ] Comment/format preservation: text surgery in the `pricestoml` style — user comments and ordering survive; idempotent (`--apply` twice ⇒ second is a no-op, byte-identical file).
- [ ] Bundled `data/policy.toml` never written; global `~/.cage/policy.toml` supported the same as project (the resolver's active policy is the target; `--ledger`/`CAGE_BASE` respected).
- [ ] Wiring into existing surfaces: doctor's and (if landed) pricing-freshness's "run cage prices sync" recommendation becomes "run `cage policy sync`" where the drift is non-price props, keeping `cage prices sync` hints for pure price drift; `cage query policy-sync` explains categories + the neutrality invariant with live values.
- [ ] Typed `CageError` on unparseable project policy (with the corrupt-file message pointing at the file); read-side only otherwise — no ledger interaction.
- [ ] Tests: each category golden; customized-never-touched; orphan-never-deleted; idempotent apply; neutrality invariant; pre-`policy_version` fallback; global-ledger target; delegation summary matches a direct `prices sync` run. Dummyrepo scenario: old-shaped policy → sync → categorized output exact → apply → derived views byte-identical. `just test` green; skillgen regen if CLI text changed.
- [ ] Docs: CHANGELOG in-tree; `docs/pricing.md` cross-link + a short section in the plan doc (§3 policy area); README one-liner; CLAUDE.md edit proposed-not-applied. Nothing committed.

## 3. Scope

**In scope:** the verb pair (`policy sync`/`policy diff`), `policy_version` meta, delegation, hint rewording, docs/query.
**Out of scope (explicit):** auto-apply anywhere (import/setup/hooks never run it); merging *user* policies across machines (fleet policy distribution is a different problem — bundles carry data, not policy); schema validation beyond what `policy.load` already tolerates; rewriting user comments/ordering; network anything.

## 4. Current state

Read first: `cage/pricescmd.py` + `cage/pricestoml.py` (the sync/customized-detection/managed-write machinery this generalizes — reuse, don't fork), `cage/policy.py` (`load`, defaults, `[meta]`), `cage/cfgio.py`, `cage/initcmd.py` (where meta gets stamped), `cage/doctorcmd.py` (recommendation strings), `data/policy.toml` (the source of truth being synced *from*), `cage/explain_data.py`, `tools/dummyrepo/run.py`, `CLAUDE.md`.

## 5. Technical approach (decided)

A `policysync.py` module: build the bundled default tree (parse `data/policy.toml` via importlib.resources), the project tree, and the old-bundled tree (reconstructed from `policy_version` — ship a small versioned-defaults record the same way prices sync knows old prices; if that record doesn't exist yet for prices, mirror whatever mechanism it actually uses — follow, don't invent). Diff into the four categories; render; `--apply` routes non-pricing writes through extended `pricestoml`-style surgery and pricing tables through `prices sync`'s apply path. CLI registers a `policy` group (future home for more).

## 6. Non-negotiables / constraints

Never auto-applied · customized values never touched, orphans never deleted · dry-run default, idempotent apply · behavior-neutrality invariant holds and is tested · bundled policy read-only · comment-preserving surgery, never whole-file rewrite · one merge brain per table family (delegate) · $0/stdlib · typed errors read-side · skillgen for rendered assets · no commits/tags/publishes.

## 7. Dependencies & prerequisites

Sequencing: `prices-route-tool` and `pricing-freshness` pairs are pending in this tree — run those first (this work rewords a hint the freshness pair introduces and delegates to machinery route-tool extends). Executor STOPs and asks if they haven't landed.

## 8. Edge cases & risks

Project policy with sections cage doesn't know (user's own keys) → untouched, not orphan-warned (orphan = bundle *used to ship it*); TOML with exotic formatting the surgery can't safely edit → per-file refusal with a clear message, never a mangled write; `--apply` on a read-only file → typed error, no partial write (write to temp + atomic replace); project == bundle exactly → "already in sync", exit 0; running inside a repo whose policy.toml is git-tracked → note in output that the diff is reviewable via git (no .bak files — git is the backup, stated).

## 9. Testing & validation

Per §2. The neutrality invariant and the idempotent-apply byte-identity are the two tests that must never be skipped — they are the safety story.
