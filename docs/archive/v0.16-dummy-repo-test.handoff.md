# Handoff: Cage end-to-end validation on a disposable repo (CLI + VS Code)

**One-liner:** Stand up a throwaway git repo, drive cage across every CLI-reachable
surface and every invariant, and produce a ranked findings list of what needs fixing —
with the VS Code-extension half handed back to a human because no agent can drive it.
**Owner / executor:** Claude Code (CLI + invariant half) · Arpit (VS Code-extension half)
**Status:** Ready to build
**Stress-tested:** Debated devils-advocate this session. Challenged: "test every feature"
as the shape. Survived: the plan does *not* re-test breadth — `just test` owns that; the
agent spends effort on the CLI-vs-VS-Code capture seam and on invariants (fail-open,
determinism, `method` integrity, PII, $0/stdlib) that a green suite still hides.
**Residual risk:** (1) The VS Code-extension path (`docs/dummy-repo-test-plan.md` §3-CLI,
§4, §7 live-agent steps) **cannot be automated by any agent** — Claude Code can't run your
extensions any more than I can. It is scoped OUT and returned to Arpit as a checklist.
(2) If the four agents aren't installed as CLIs on the target machine, real capture (§3)
can't run; the agent must fall back to **seeding a synthetic ledger** so §5/§6 still
execute. This branch is pre-authorized below, not an open question.

## 1. Context & background
Cage is a $0, stdlib-only, deterministic attribution ledger for LLM token traffic
(`/Users/arpitarya/my_programs/cage`, `cage 0.15.0`). It supports four agents
(claude/codex/copilot/kiro) as a product invariant. The full manual plan already exists at
`docs/dummy-repo-test-plan.md`; this handoff packages the **agent-executable subset** of
it. The point of the exercise is not to prove features render — the test suite does that —
but to catch what breaks at the integration seams and under adversarial input.

## 2. Definition of done
The run is done when ALL are true:
- [ ] `just test` was run from the cage repo and the pass/fail count is recorded.
- [ ] A disposable repo exists at `/tmp/cage-testbed` with `.cage/` scaffolded and at
      least one commit.
- [ ] The ledger is populated — by **real CLI capture** if any agent CLI is present, else
      by a **synthetic seed** (documented as such in the findings).
- [ ] Every read/derive surface in §5 of the plan has been run and its outcome recorded
      (clean exit vs typed error vs crash).
- [ ] Every invariant/adversarial check in §6 of the plan has been run with a pass/fail
      verdict and evidence.
- [ ] Provenance/git-hook checks (§7) that are CLI-reachable have been run.
- [ ] A findings table (plan §8 schema) is written to
      `/tmp/cage-testbed/CAGE-FINDINGS.md`, ranked P0/P1/P2.
- [ ] The VS Code-extension steps are listed as an explicit **human checklist** in the
      findings file, marked `NOT RUN — requires live extension`.
- [ ] Documentation impact assessed (see §9.5).

## 3. Scope
**In scope (Claude Code executes):**
- Baseline `just test`.
- Scaffold `/tmp/cage-testbed` and wire agents via `cage setup`.
- CLI capture (§3 of plan) **only for agents installed as CLIs on this machine**.
- Synthetic ledger seed if no agent CLI is available.
- All read/derive surfaces (§5), all invariant/adversarial checks (§6), CLI-reachable
  provenance/git-hook checks (§7).
- Produce `CAGE-FINDINGS.md`.

**Out of scope (explicit — do NOT attempt):**
- Running any agent **as a VS Code extension**. Claude Code cannot drive extensions;
  do not fake, mock, or simulate an "extension run" and report it as tested. List these
  steps as a human checklist instead.
- **Fixing** any bug found. This task is validation only — findings, not patches. A
  separate handoff covers fixes.
- Editing cage source, `policy.toml` defaults, or the test suite to make a check pass.
- Writing to `refs/notes/*` (notes-sync/ledger-sync must stay dry-run on a dev machine).

## 5. Technical approach (decided)
- **Reuse the existing plan verbatim** — do not redesign it. Execute plan §1, §2, §5, §6,
  and the CLI-reachable parts of §3/§7. Skip §4 and live-extension §3/§7 (out of scope).
- **Capture branch:** probe for agent CLIs first (`cage setup --status`, `which claude`
  etc.). If ≥1 present, do real CLI capture. If none, seed the ledger synthetically via
  `cage`'s own demo/seed path (`cage demo` seeds the plan §4.4 rows) plus a couple of
  hand-crafted rows to exercise pricing/human/unpriced-model cases — and label every
  synthetic-derived finding as such.
- **Evidence, not assertion:** every §6 verdict must carry the command output or a diff,
  not just "pass." Determinism = an actual byte-diff of two runs; PII = the actual grep
  output; fail-open = the actual return on a truncated shard.

## 6. Non-negotiables / constraints
- **Read-only on cage itself.** Do not modify cage source, the bundled `policy.toml`, or
  `tests/`. All mutation happens inside `/tmp/cage-testbed` or a throwaway `CAGE_BASE`.
- **Never simulate the un-runnable.** If a step needs a VS Code extension or an
  uninstalled agent, mark it `NOT RUN` with the reason — never fabricate a result.
- **Preserve cage's own rules while testing:** stay offline where the plan says offline
  (proves $0/stdlib), keep `notes-sync`/`ledger-sync` dry-run, don't set
  `CAGE_NOTES_WRITE`.
- **Determinism discipline:** when diffing two runs for determinism, hold policy and
  ledger fixed between them; don't let a capture fire between the two reads.
- **Do not touch:** `~/.claude`, `~/.codex`, `~/.copilot`, `~/.kiro` contents (read for
  diagnosis only); the user's real `~/.cage` global ledger (use an isolated `CAGE_BASE`
  or the project `.cage/`).

## 7. Dependencies & prerequisites
- cage on PATH (`cage --version` → 0.15.0); Python with no extras installed.
- `git` available; a writable `/tmp`.
- Optional: any of the four agent CLIs. Absence is handled by the synthetic-seed branch —
  not a blocker.
- No secrets required. If `CAGE_HUMAN_RATE` is exercised, set it inline for that step
  only; never persist.

## 8. Edge cases & risks
- **No agent CLI installed** → synthetic-seed branch (§5); mark findings synthetic.
- **`policy.toml` lacks `anthropic` model rows** → Claude rows price at $0. This is itself
  a finding to test for (plan §6 unpriced-model), not something to silently patch.
- **Truncated/torn ledger tail** → reads must tolerate it; if a read raises, that's a P0.
- **Non-repo / detached HEAD** during provenance checks → git fields omitted, no crash.
- **`cage verify` must exit 0 always** — a non-zero exit is a P0 finding.
- **Global-ledger contamination** → always isolate via project `.cage/` or a scratch
  `CAGE_BASE`; verify the active ledger with `cage doctor` before generating traffic.

## 9. Testing & validation
- The deliverable *is* a test pass; "tests" here = the plan's checklists.
- Verify the harness itself: `cage --version`, `cage doctor` (active ledger correct),
  `python -c "import cage"` offline.
- `just test` from the cage repo is the breadth baseline; record its result first.
- Final self-check: re-open `CAGE-FINDINGS.md` and confirm every plan §5/§6 row has a
  verdict and every out-of-scope step is marked `NOT RUN` with a reason.
